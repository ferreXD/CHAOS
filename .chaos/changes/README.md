# CHAOS Changes — v0 team-safe collaboration model

> Canonical contract for how change-scoped CHAOS artifacts are laid out, named, and
> synced. Generated by `chaos:init --guided` on 2026-07-19. `chaos:init` scaffolds this
> workspace but does **not** create per-change folders — commands create those lazily.

## Per-change layout

Every change owns one folder, keyed by its OpenSpec change id:

```text
.chaos/changes/<change-id>/
  lifecycle.md          # status + links tying the change's artifacts together (the manifest)
  proposal-report.md    # chaos:propose
  proposal-review.md    # chaos:review
  approval.md           # chaos:review (approval gate)
  apply-report.md       # chaos:apply (+ scope-drift assessment)
  verification.md        # chaos:verify (spec-traceability matrix)
  archive-report.md     # chaos:archive
  sync-report.md        # chaos:sync --change
  decision-events.md    # PROP-DEC-* / REV-DEC-* / APP-DEC-* — the human decision record
  waivers.md            # accepted risk / debt
```

The OpenSpec source of truth for the same change lives in parallel under
`openspec/changes/<change-id>/` (proposal, design, specs, tasks). CHAOS **links** to it;
it never copies between the two trees.

## Lifecycle manifest (`lifecycle.md`)

Each change's `lifecycle.md` records the current lifecycle state (proposed → reviewed →
applied → verified → archived), the mode (`--light`/`--standard`/`--strict`), and links
to every artifact above. It is the single place to see where a change stands.

## Artifact naming policy

- Physical filenames within a change folder use the fixed names above.
- Repository-wide, non-change-scoped reports use **date-prefixed slug** filenames
  (`YYYY-MM-DD-<slug>-<kind>.md`), e.g. `.chaos/sync-reports/repo-sync-2026-07-19.md`.
- **Sequential IDs** (e.g. decision IDs) are assigned in **indexes only**, never encoded
  into physical filenames. `dateFormat: YYYY-MM-DD`; a slug is required.

## Team concurrency policy

- Multiple contributors may work on **different** changes concurrently — each change
  folder is independent, and the interaction runtime holds a **change lock** while a
  command has an open decision, so two commands cannot mutate the same change at once.
- Do not hand-edit another contributor's in-flight change folder; drive it through the
  commands so the audit trail stays intact.

## `chaos:sync` role model

| Scope | Command | Who |
|---|---|---|
| One change | `chaos:sync --change <change-id>` | contributor-safe |
| By ref / ADRs / rules / gates / agents | `chaos:sync --since <ref>` / `--adrs` / `--rules` / `--gates` / `--agents` | maintainer |
| Repo-wide | `chaos:sync --all` → `.chaos/sync-reports/repo-sync-YYYY-MM-DD.md` | repo-owner only (requires confirmation) |

After a change merges into `main`, a **mainline sync** is recommended
(`policies.sync.mainlineBranch: main`). Repo-wide sync requires provider/repository
context and maintainer/repo-owner confirmation (`policies.repositoryContext`).

Config anchors: `.chaos/config.yaml` → `paths.changes`, `policies.changeArtifacts`,
`policies.sync`, `policies.artifactNaming`.
