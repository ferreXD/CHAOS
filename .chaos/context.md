## Project summary

**Task Tracker API** is a small, fully runnable ASP.NET Core **Minimal API** with an
in-memory task store and plain CRUD over `/tasks`. It is the runnable subject that the
CHAOS worked example drives through the lifecycle (propose → review → apply → verify →
archive → sync). `[FACT]` — inspected `src/TaskTracker.Api/**`, `TaskTracker.sln`,
`examples/task-tracker/dotnet/README.md`.

The API works today, but **`GET /tasks` has no filtering**. Adding optional `?status=`
and `?priority=` filters is the deliberate gap the demo closes. `[FACT]` — see
`Endpoints/TaskEndpoints.cs` and the demo README.

## Domain summary

A **task** is an immutable record: `Id` (GUID), `Title`, `Status`, `Priority`,
`CreatedAt`. Status is `Open | InProgress | Done` (enum `TaskState`); priority is
`Low | Medium | High` (enum `TaskPriority`). Both serialize as their names. `[FACT]` —
`Domain/TaskItem.cs`, `Program.cs` (`JsonStringEnumConverter`).

> **Naming note:** the status enum is `TaskState`, not `TaskStatus`, to avoid colliding
> with `System.Threading.Tasks.TaskStatus` under implicit global usings. `[FACT]`

## Important flows

| Flow | Route(s) | Notes |
|---|---|---|
| Health check | `GET /` | `{ "service": "task-tracker", "status": "ok" }` `[FACT]` |
| List tasks | `GET /tasks` | Returns **all** tasks, unfiltered (the demo gap) `[FACT]` |
| Get one | `GET /tasks/{id:guid}` | `200` or `404` `[FACT]` |
| Create | `POST /tasks` | `201 Created`; `400` if `Title` blank `[FACT]` |
| Replace | `PUT /tasks/{id:guid}` | `200` or `404`; `400` if `Title` blank `[FACT]` |
| Delete | `DELETE /tasks/{id:guid}` | `204` or `404` `[FACT]` |

## Constraints

- Targets `net8.0` (builds on newer SDKs). Requires .NET SDK 8.0+. `[FACT]` — `.csproj`.
- In-memory store (`ConcurrentDictionary`, singleton, seeded) — **state resets on every
  restart**; no persistence layer. `[FACT]` — `Domain/TaskStore.cs`.
- `Nullable` and `ImplicitUsings` enabled; `InvariantGlobalization` true. `[FACT]`
