# chaos:status Check Catalog

## Structure checks

### CS-STRUCT-001 — Required root files

Required:

- `AGENTS.md`
- `README.md` unless explicitly skipped
- `.chaos/config.yaml`
- `.chaos/context.md`
- `.chaos/architecture.md`
- `.chaos/constitution.md`
- `.chaos/bootstrap-report.md`
- `.chaos/decisions/index.md`
- `.chaos/rules/index.md`
- `.chaos/commands/index.md`
- `.chaos/gates/index.md`

Failure severity:
- Missing `AGENTS.md`: BLOCKER
- Missing `.chaos/bootstrap-report.md`: BLOCKER
- Missing `.chaos/config.yaml`: MAJOR by default for config-aware workspaces; BLOCKER in `--strict` unless explicitly deferred
- Missing any `.chaos/*/index.md`: MAJOR or BLOCKER depending on strict mode
- Missing README: MINOR unless README generation was required by init

### CS-STRUCT-002 — Folder-index model

`decisions`, `rules`, `commands`, and `gates` must be folders with `index.md`, not only raw top-level files.

- `.chaos/architecture.md`

### CS-AGENTS-002 — AGENTS.md is not bloated

## Bootstrap audit checks

### CS-BOOT-001 — Bootstrap report exists

`.chaos/bootstrap-report.md` is mandatory.

### CS-BOOT-002 — Mode is recorded

The report must record:

- default
- `--auto`
- `--guided`

### CS-BOOT-003 — Questions and answers are recorded

The report must record questions asked and user answers, or explicitly state that no questions were asked because mode was `--auto`.

### CS-BOOT-004 — Scope decisions are explicit

The report must clearly list included and excluded scopes.

### CS-BOOT-005 — Major track exclusions are confirmed

If docs exist for a major track but the track is excluded, there must be explicit user confirmation.

### CS-BOOT-006 — Proposed ADR treatment is explicit

If Proposed/Draft ADRs are treated as accepted working posture, the report must record explicit user confirmation.


## Config checks

### CS-CONFIG-001 — Config presence

`.chaos/config.yaml` should exist in config-aware workspaces generated by current `chaos:init`.

Failure severity:
- MAJOR in default mode if config is missing and no explicit deferral exists.
- BLOCKER in `--strict` mode.

### CS-CONFIG-002 — Config version is supported

`version` must exist and be compatible with this command. v0 expects `0.1`.

Unsupported versions produce `CONFIG_UNSUPPORTED_VERSION` and must not be silently interpreted.

### CS-CONFIG-003 — Required sections are present

Expected top-level sections:

- `version`
- `project`
- `paths`
- `agents`
- `toolchain`
- `validation`
- `policies`

Missing material sections produce `CONFIG_PARTIAL`.

### CS-CONFIG-004 — Config remains a conventions file

Config must not contain secrets, credentials, connection strings, architecture decisions, giant rule/gate/prompt bodies, or hidden approval switches.

Secrets or hidden bypasses are MAJOR by default and BLOCKER in `--strict` mode.

### CS-CONFIG-005 — Path coherence

Configured paths must be coherent with discovered repository files. Missing configured paths must be reported as missing, not silently treated as valid.

### CS-CONFIG-006 — Toolchain and validation commands are sourced honestly

Toolchain and validation commands should come from `.chaos/config.yaml` when present. If not executed, mark them as configured but unverified.

### CS-CONFIG-007 — Protected file policy exists

Config should describe protected-file handling for `AGENTS.md` and root `README.md`, including whether status/sync may edit them and whether patch preview is required.

### CS-CONFIG-008 — Bootstrap report records config handling

`.chaos/bootstrap-report.md` should record whether config was created, preserved, patched, conflicted, or deferred during init.

## Change-scoped layout & collaboration checks (v0)

Canonical contract: `.chaos/changes/README.md`. These are read-only audits; do not mutate
shared files except through the existing confirmed remediation flow.

### CS-CHG-001 — Config declares the change-scoped layout

`.chaos/config.yaml` should declare `paths.changes` (`.chaos/changes`) and
`policies.changeArtifacts.layout` = `.chaos/changes/<change-id>`, with
`writePerChangeReportsUnderChangeFolder: true`, `readLegacyReportFolders: true`, and
`migrateLegacyReportsAutomatically: false`.

Failure severity: MAJOR by default; advisory in `--light`.

### CS-CHG-002 — Config declares the artifact-naming policy

`.chaos/config.yaml` should declare `policies.artifactNaming` with
`physicalFilenamesUseDatePrefix: true` and `sequentialIdsAssignedInIndexesOnly: true`.

Failure severity: MAJOR by default.

### CS-CHG-003 — Config declares the sync role/collaboration policy

`.chaos/config.yaml` should declare `policies.sync` with
`changeScopedAllowedForContributors: true`,
`repoWideSyncRequiresMaintainerConfirmation: true`, and a `mainlineBranch`.

Failure severity: MAJOR by default.

### CS-CHG-004 — No legacy-only output contracts (drift)

Detect command contracts that still point **only** to legacy scattered folders
(`.chaos/reviews/`, `.chaos/apply-reports/`, `.chaos/verification/`,
`.chaos/archive-reports/`, `.chaos/retros/`) as the preferred output, with no per-change
(`.chaos/changes/<change-id>/`) target. Report as `LEGACY_ONLY_OUTPUT_CONTRACT` drift.

Failure severity: MAJOR by default; BLOCKER in `--strict` if it would misroute new
artifacts.

### CS-CHG-005 — Commands index documents the new layout

`.chaos/commands/index.md` should document the per-change artifact layout and the
distinction between `chaos:sync --change <id>` (contributor-safe) and `chaos:sync --all`
(maintainer/repo-owner). Missing documentation is `LAYOUT_DOC_DRIFT`.

### CS-CHG-006 — Concurrency documentation present

`AGENTS.md`, root `README.md`, and `.chaos/README.md` should mention the team concurrency
policy and mainline sync policy. Missing mentions are `CONCURRENCY_DOC_DRIFT` (route through
the protected-doc remediation flow; do not edit silently).

### CS-CHG-007 — Legacy artifacts are read-compatible, not migrated

If legacy scattered report folders exist alongside `.chaos/changes/`, status may recommend a
future migration but must not perform or require it. Report as informational
`LEGACY_LAYOUT_PRESENT`.

## Claude command-suite hardening checks (model robustness)

These checks audit whether the Claude command surface declares the model-robustness
hardening that lets CHAOS run reliably on the weakest supported Claude model. They are
read-only audits over `.claude/commands/*.md`, `.claude/skills/*/**`, and
`.claude/agents/*.md`. Report missing items as workflow drift; remediation routes to
`chaos:sync` (`CS-HARDEN-*` reconciliation) with patch preview and confirmation. Do not
edit `.claude/` contracts silently.

Canonical policy: `.claude/skills/chaos-shared/reference/model-robustness-policy.md`.

### CS-HARDEN-001 — Shared model-robustness policy present

`.claude/skills/chaos-shared/reference/model-robustness-policy.md` and
`interactive-decision-protocol.md` exist.

Failure severity: MAJOR by default; BLOCKER in `--strict`.

### CS-HARDEN-002 — Command execution contracts present

Every `.claude/commands/chaos-*.md` (aliases may inherit by reference) declares a compact
"Non-negotiable execution contract" near the top, referencing the shared policies.

Failure severity: MAJOR by default; advisory in `--light`. Report missing commands as
`MISSING_EXECUTION_CONTRACT`.

### CS-HARDEN-003 — OpenSpec invocation gate in chaos:propose

`chaos:propose` contracts (command, skill, `openspec-integration-contract.md`, architect
agent) declare the hard OpenSpec invocation gate and require an **OpenSpec Invocation
Proof** in the report. A `chaos:propose` that can silently bypass OpenSpec when OpenSpec is
available is `MISSING_OPENSPEC_GATE`.

Failure severity: BLOCKER in `--strict`; MAJOR by default.

### CS-HARDEN-004 — Stop-after-decision requirement

Command wrappers state that the command STOPS after presenting a material decision and does
not continue until the user selects an option. Missing is `MISSING_STOP_AFTER_DECISION`.

Failure severity: MAJOR by default.

### CS-HARDEN-005 — Change-scoped output layout declared

Change-scoped commands (`propose`, `review`, `apply`, `verify`, `archive`,
`sync --change`, `retro`) declare `.chaos/changes/<change-id>/` as the new-artifact target.
Reuses `CS-CHG-004` detection for legacy-only contracts.

Failure severity: MAJOR by default.

### CS-HARDEN-006 — Sync role boundaries + maintainer gate

`chaos:sync` contracts declare contributor-safe vs maintainer vs repo-owner scopes and the
exact `--all` maintainer confirmation prompt (cross-checked with `CS-SYNCPOSTURE-002`).
A weakened or bypassable gate is `WEAKENED_SYNC_AUTHORITY`.

Failure severity: MAJOR by default; BLOCKER in `--strict`.

### CS-HARDEN-007 — Date-prefixed artifact-naming policy referenced

Command/skill contracts reference the artifact-naming policy (date-prefixed physical
filenames; sequential IDs display-only). Missing is `MISSING_ARTIFACT_NAMING_REF`.

Failure severity: MINOR by default; MAJOR in `--strict`.

### CS-HARDEN-008 — Interactive decision protocol referenced

Command wrappers reference the interactive decision protocol (numbered options, recommended
option marked, native selection UI preferred with numbered fallback). Missing is
`MISSING_DECISION_PROTOCOL_REF`.

Failure severity: MAJOR by default.

## Repository-wide sync posture checks

### CS-SYNCPOSTURE-001 — Repo-wide sync recency

Report the most recent `chaos:sync --all` (`.chaos/sync-reports/repo-sync-YYYY-MM-DD.md`)
and/or `--since main` run. If feature branches appear merged into `main` without a
subsequent repository-wide reconciliation, recommend `chaos:sync --all` (run by the repo
owner / CHAOS maintainer). This is posture reporting, not a blocker by default.

### CS-SYNCPOSTURE-002 — Maintainer-confirmation policy intact

Verify config/contracts still require maintainer confirmation for `chaos:sync --all`
(except `--dry-run`). A weakened or bypassable gate is a MAJOR governance finding.

## Source inventory checks

### CS-SRC-001 — Exact source manifest

ADR-like sources must be listed individually with path, ID, title, detected status, inventory status, scope inclusion, and reason.

### CS-SRC-002 — No unsafe range summaries

Source lists must not use incomplete ranges like `ADR-0001...ADR-0013` unless missing items are explicitly listed.

### CS-SRC-003 — Verified/missing/inferred status

Every source must be marked as one of:

- verified
- missing
- inferred

### CS-SRC-004 — Source claims are path-validated

A file path claim is `verified` only if the file exists.

## Decision checks

### CS-DEC-001 — Decisions index exists and is useful

`.chaos/decisions/index.md` must group decisions by topic and link to sources.

### CS-DEC-002 — Decision statuses are not flattened

Proposed, accepted, deferred, superseded, rejected, and unknown statuses must remain visible.

### CS-DEC-003 — Operational consequences are present

Each major decision area should include operational consequences for agents.

## README checks

### CS-README-001 — README exists or is explicitly skipped

If commands cannot be verified, README should state that they are unknown.

### CS-README-003 — README links to CHAOS workspace

## Readiness checks

### CS-READY-001 — Next command is identifiable

The status report must recommend the next CHAOS command or state that no next command is safe.

### CS-READY-002 — Blockers prevent readiness

If any BLOCKER finding exists, verdict cannot be READY or STRONG.

### CS-READY-003 — Unknowns affect verdict honestly

If critical unknowns exist, verdict must be NEEDS_ATTENTION or BLOCKED depending on severity.

## Constitution checks

### CS-CONST-001 — Confidence doctrine exists

`.chaos/constitution.md` must require material findings, recommendations, approvals, verifications, gate results, and verdicts to declare knowledge type and confidence.

Required knowledge types:

- `FACT`
- `INFERENCE`
- `ASSUMPTION`
- `UNKNOWN`
- `CONFLICT`

Required confidence levels:

- `HIGH`
- `MEDIUM`
- `LOW`

Failure severity:

- MAJOR by default.
- BLOCKER in `--strict` mode.
- BLOCKER if `chaos:verify`, `chaos:approve`, or any gate command is marked implemented but the constitution lacks this doctrine.

### CS-CONST-002 — No confidence-less verdicts principle

The constitution must explicitly prohibit:

- confidence-less verdicts;
- unlabeled assumptions;
- inference disguised as fact;
- silent conflict resolution.

Failure severity: MAJOR.

### CS-CONST-003 — Verdict metadata requirement

The constitution must require final verdicts to include:

- confidence;
- evidence coverage;
- assumption load.

Failure severity: MAJOR.

### CS-CONST-004 — Remediation prompt required

If CS-CONST-001, CS-CONST-002, or CS-CONST-003 fails, `chaos:status` must ask the user whether to:

1. patch `.chaos/constitution.md` now;
2. create a remediation plan;
3. defer with rationale;
4. mark the risk as accepted;
5. do nothing.

The command must not patch without explicit user confirmation.


## Toolchain checks

### CS-TOOL-001 — Git is available

Run or request `git --version`.

Failure severity:
- FAIL: MAJOR by default; BLOCKER in strict mode if source manifests depend on Git history.

### CS-TOOL-002 — Node.js supports OpenSpec

Run or request `node --version`. Node.js must be `>= 20.19.0` for the OpenSpec CLI workflow.

Failure severity:
- Missing Node.js: MAJOR.
- Node.js below minimum: MAJOR, BLOCKER for OpenSpec-dependent commands.

### CS-TOOL-003 — npm is available

Run or request `npm --version`.

Failure severity:
- MAJOR when OpenSpec is expected.

### CS-TOOL-004 — OpenSpec CLI is available

Run or request `openspec --version`.

Failure severity:
- MAJOR when the workspace declares OpenSpec as the spec motor.
- BLOCKER for `chaos:propose`, `chaos:apply`, `chaos:archive`, and OpenSpec-backed validation readiness unless explicitly deferred.

### CS-TOOL-005 — Tool remediation is human-approved

If a required tool is missing, `chaos:status` must ask the user whether to install, show manual install instructions, continue in degraded mode, defer, or abort.

No tool may be installed silently.


### CS-AGENTS-003 — AGENTS.md is current

`AGENTS.md` / `AGENT.md` must not contradict current CHAOS indexes or installed command artifacts.

Detect stale:

- rule ranges;
- gate ranges;
- command lists;
- command implementation status;
- protected-file edit policy;
- references to missing `.chaos` files.

Drift severity:

- MAJOR by default.
- BLOCKER in `--strict` if the drift would misroute agents or hide mandatory rules/gates.

### CS-AGENTS-004 — Singular/plural agent entrypoint conflict

If both `AGENT.md` and `AGENTS.md` exist, report whether one is canonical or whether they conflict. Prefer `AGENTS.md` as the canonical multi-agent repository entrypoint unless config explicitly says otherwise.

## Protected documentation checks

### CS-PDOC-001 — Protected docs are protected-but-editable

`.chaos/config.yaml` may require confirmation and patch preview for `AGENTS.md` / `AGENT.md` and root `README.md`, but status/sync must be allowed to propose and apply confirmed patches or rewrites.

If policy blocks all edits, report `PROTECTED_DOC_POLICY_TOO_STRICT` and recommend either:

- one-time protected-doc override with rationale;
- config policy update allowing confirmed status/sync edits.

### CS-PDOC-002 — Protected doc patch preview required

Any remediation that changes `AGENTS.md`, `AGENT.md`, or `README.md` must show a patch or rewrite preview before writing.

### CS-PDOC-003 — Protected doc remediation is auditable

The status report must record the protected-doc finding, recommended action, user decision, files changed, and whether override was used.

### CS-README-004 — README is current with CHAOS indexes

README must not contain stale command counts, stale rule/gate ranges, obsolete lifecycle descriptions, or contradicted command semantics.

### CS-README-005 — README protected edit policy is clear

README should explain that generated governance summaries may be patched by `chaos:status` or `chaos:sync` only with patch preview and explicit confirmation.
