# CHAOS Todo Write Policy

## Allowed writes

`chaos:todo` may write only:

```text
.chaos/todo/items/*.md
.chaos/todo/index.md
.chaos/todo/views/*.html
.chaos/todo/todo-report-YYYY-MM-DD.html
.chaos/changes/<change-id>/todo-candidates.md   # only if explicitly invoked with --from-change and the user opts into a change-scoped candidate snapshot
```

## Forbidden writes

`chaos:todo` must never write:

```text
production source code
tests
migrations
OpenSpec artifacts (openspec/**)
ADR files (docs/adr/**)
decision logs (docs/decision-log/**)
.chaos/rules/**
.chaos/gates/**
AGENTS.md
README.md
root project docs
```

and must not mutate existing roadmap items, existing CHAOS reports, or other commands' report
files — it may only *reference* them.

## Evidence requirement (`requireSourceEvidence: true`)

A durable todo item may be created or kept `open` only if it resolves to at least one
existing `sourceArtifacts` path (ideally with `sourceIds` inside that artifact). No
evidence-free items. If a previously-open item's source artifact no longer exists or no longer
contains the referenced finding, `chaos:todo --strict` flags it `stale` (see
`todo-status-model.md`) rather than silently keeping it `open`.

## Deduplication before write (`dedupeBeforeWrite: true`)

Always run `todo-deduplication-policy.md` before creating a new item or updating an existing
one. Never write two items for the same underlying capability gap.

## Default write mode: confirm

`chaos:todo` without `--write` never writes durable files. It shows the dashboard
(`todo-report-template.md`) and asks:

```text
Write/update todo backlog and HTML views?

1. Yes, write proposed updates
2. Dry-run only
3. Show details
4. Stop
```

Stop after presenting this. `--write` skips this specific prompt (but not maintainer-sensitive
prompts below). `--dry-run` never writes, even if `--write` is also passed — `--dry-run` wins.

## Contributor-safe vs maintainer-sensitive

| Contributor-safe (no extra gate) | Maintainer-sensitive (requires the maintainer confirmation decision) |
|---|---|
| `chaos:todo` | `chaos:todo --all --write` (there is no `--all`; repository-level writes without a narrowing scope) |
| `chaos:todo --next` | `chaos:todo --from-roadmap --write` |
| `chaos:todo --from-change <change-id>` | `chaos:todo --dedupe --write` |
| `chaos:todo --scan --dry-run` | `chaos:todo --close <todo-id>` for a repository-level (non-`current-change`-scoped) todo |
| `chaos:todo --from-change <change-id> --write` | `chaos:todo --refresh` when it regenerates repository-wide HTML views |

For maintainer-sensitive writes, ask exactly this decision and STOP:

```text
Decision required: Repository-level todo update

Context:
This operation may update repository-level todo items, indexes, and generated HTML views.

Recommended option:
1. Continue only if you are acting as repository owner or designated CHAOS maintainer.

Options:
1. Yes, continue and record maintainer confirmation.
2. Switch to dry-run only.
3. Limit to a specific change id.
4. Stop.

Select one option to continue.
```

Record the selected option as a Context Note (or Decision Event, when a change-scoped
lifecycle exists) in the write summary.

## Regeneration policy for HTML views

- HTML views are regenerated by running the generator (`node tools/chaos-todo-views/generate.mjs`)
  on `chaos:todo --refresh`, or automatically after a write that changes item files
  (create/update/close/reopen), subject to `policies.todo.htmlViews.regenerateOnlyWhenSourcesChange`.
- HTML views must never be hand-edited; if a view is stale relative to its source item files,
  regenerate it rather than patch it.
- Do not regenerate HTML on every invocation when no source item file changed and
  `regenerateOnlyWhenSourcesChange: true` (default).
