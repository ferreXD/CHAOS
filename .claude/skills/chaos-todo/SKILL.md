# CHAOS Todo Skill

Use this skill when the user invokes `chaos:todo` (or `/chaos-todo`), or asks what concrete
work is pending, what the current backlog looks like, or to turn a CHAOS report's findings
into trackable follow-up.

## Identity

CHAOS reports create evidence. `chaos:todo` turns evidence into executable backlog.

`chaos:todo` discovers, normalizes, deduplicates, prioritizes, and maintains actionable work
generated by CHAOS workflows. It is **not** a replacement for:

- `chaos:status` — is the workspace/governance healthy?
- `chaos:sync` — does governance/source-of-truth need reconciliation?
- `chaos:retro` — what did we learn from a completed change?
- roadmap files (`.chaos/roadmap/*.md`) — what release/product direction do we intend?

`chaos:todo` answers: **what concrete actionable work is pending?**

## Non-negotiable execution contract (model robustness)

Executable by the weakest supported Claude model. Obey
`.claude/skills/chaos-shared/reference/model-robustness-policy.md` and
`.claude/skills/chaos-shared/reference/interactive-decision-protocol.md`.

1. **Read-first, safe by default.** No flags → read existing index/items + latest
   status/doctor, show the dashboard, ask before writing.
2. **Evidence required.** A durable todo item requires a resolvable source artifact
   (`policies.todo.requireSourceEvidence`). No evidence-free items.
3. **Dedupe before write.** Always run deduplication before creating/updating a durable item.
4. **Stop after material decisions.** One decision at a time, native selection UI preferred,
   numbered fallback, STOP immediately after presenting it.
5. **Markdown items, HTML digests.** Durable todo items are Markdown under
   `.chaos/todo/items/`. Human digest reports/views are static, self-contained, escaped HTML —
   never Markdown digest reports.
6. **Do not over-import.** Not every warning is a todo; not every command becomes a todo
   writer. Only `chaos:todo` writes durable backlog files.
7. **Scope.** Never edit production code, tests, migrations, OpenSpec content, ADRs, decision
   logs, rules, gates, `AGENTS.md`, or `README.md`. Never mutate existing roadmap items or
   other CHAOS reports — reference them only.

## Invocation

```text
chaos:todo
chaos:todo --scan
chaos:todo --next
chaos:todo --from-roadmap
chaos:todo --from-audit <path>
chaos:todo --from-change <change-id>
chaos:todo --target <target>
chaos:todo --priority <priority>
chaos:todo --status <status>
chaos:todo --type <type>
chaos:todo --write
chaos:todo --dry-run
chaos:todo --dedupe
chaos:todo --refresh
chaos:todo --close <todo-id>
chaos:todo --reopen <todo-id>
chaos:todo --update <todo-id>
```

Modes `--light` / `--standard` (default) / `--strict` per `reference/todo-command-contract.md`.

## Workflow

1. Read `.chaos/config.yaml`; resolve `paths.todo` / `paths.todoItems` / `paths.todoViews` and
   `policies.todo.*` before defaults (`reference/todo-config-contract.md`). If missing, follow
   the mode-specific config-missing behavior.
2. Resolve mode (explicit or inferred) and invocation intent (`--scan`, `--next`,
   `--from-roadmap`, `--from-audit`, `--from-change`, filters, `--dedupe`, `--refresh`,
   `--close`/`--reopen`/`--update`).
3. Read the existing `.chaos/todo/index.md` and item files under `.chaos/todo/items/` (always,
   regardless of mode) so state is never rebuilt from scratch.
4. Scan the configured evidence sources appropriate to the mode
   (`reference/todo-source-contract.md`). `--light` skips the deep scan; `--standard` scans
   normal sources; `--strict` scans everything and also validates every open item's evidence
   (`reference/todo-status-model.md`).
5. Extract candidates, applying the materiality guardrail
   (`reference/todo-candidate-contract.md`). Also read any "## Todo Candidates" sections
   emitted by other commands' reports as an additional source.
6. Deduplicate candidates against each other and against existing items
   (`reference/todo-deduplication-policy.md`). Ask, one at a time, when a match is ambiguous.
7. If `--from-roadmap`, run the roadmap import decision
   (`reference/todo-roadmap-bridge.md`) and STOP for the user's choice before proceeding.
8. Build the chat-first dashboard (`reference/todo-report-template.md`) and show it.
9. Ask the write decision (unless `--write` or `--dry-run` already resolves it). For
   repository-level (maintainer-sensitive) writes, ask the maintainer confirmation decision
   (`reference/todo-write-policy.md`) — one decision, STOP.
10. On confirmed write: create/update/close/reopen item files under `.chaos/todo/items/`
    (date-prefixed slug filenames, never sequential — `reference/todo-item-schema.md`), update
    `.chaos/todo/index.md`, then regenerate the affected static HTML views
    (`reference/todo-html-view-contract.md`, `reference/todo-html-template.md`) unless
    `policies.todo.htmlViews.regenerateOnlyWhenSourcesChange` skips an unchanged view.
11. Summarize: counts, what was written (or would be written, in dry-run), and the recommended
    next command.

## `--next`

Read-only. Show the dashboard's "Recommended next" list from the existing index/items (plus a
light top-up scan of `.chaos/status-report.md` / latest doctor report) without a full re-scan.
Useful for `chaos:help next` to consult.

## `--from-audit <path>`

Treat `<path>` as a single evidence artifact to scan (in addition to, not instead of, the
normal source set unless the user says otherwise). Useful for a one-off report not under a
configured default path.

## `--from-change <change-id>`

Scan only `.chaos/changes/<change-id>/*.md` (apply/verify/review/archive/retro/decision-events/
waivers/todo-candidates). Contributor-safe; does not require the maintainer confirmation gate
unless it also touches repository-level items.

## `--dedupe`

Re-run deduplication across all existing open items (not just newly scanned candidates).
Repository-wide `--dedupe --write` is maintainer-sensitive.

## `--refresh`

Regenerate all HTML views from current item files/index without rescanning evidence sources.
Repository-wide `--refresh` is maintainer-sensitive per `reference/todo-write-policy.md`.

## `--close <todo-id>` / `--reopen <todo-id>` / `--update <todo-id>`

Status-model operations — see `reference/todo-status-model.md`. Closing/reopening/updating a
repository-level todo is maintainer-sensitive; a `current-change`-scoped todo tied to the
user's own active change is contributor-safe.

## Output

```text
.chaos/todo/items/YYYY-MM-DD-<slug>.md     # durable source of truth
.chaos/todo/index.md                       # durable repository index (Markdown)
.chaos/todo/views/index.html               # generated digest (static HTML)
.chaos/todo/views/public-alpha.html
.chaos/todo/views/v1.html
.chaos/todo/views/open-blockers.html
.chaos/todo/views/by-type.html
.chaos/todo/views/by-change.html           # optional
.chaos/todo/todo-report-YYYY-MM-DD.html    # dated snapshot digest
```

## Contributor vs maintainer-sensitive

See `reference/todo-write-policy.md` for the full table. In short: single-change-scoped,
dry-run, and read-only invocations are contributor-safe. Repository-wide writes, roadmap
imports that write, `--dedupe --write`, closing a repository-level todo, and repository-wide
`--refresh` require the maintainer confirmation decision.

## Do not

- Do not create durable items without source evidence.
- Do not skip deduplication before writing.
- Do not generate Markdown digest reports — human digests are HTML only.
- Do not hand-edit generated HTML views.
- Do not use sequential IDs as physical item filenames.
- Do not edit production code, tests, migrations, OpenSpec content, ADRs, decision logs, rules,
  gates, `AGENTS.md`, or root `README.md`.
- Do not mutate roadmap files or other CHAOS reports — reference their IDs only.
- Do not implement remote GitHub Issues / Azure Boards export or work-item synchronization.
- Do not promote every runtime hook/decision-wait event automatically.
- Do not turn every command into a todo writer; only `chaos:todo` owns `.chaos/todo/items/`.

## Repository context (optional)

When easily available, `chaos:todo` may record repository provider/branch context
(`.claude/skills/chaos-shared/reference/repository-context-contract.md`) in the dashboard/HTML
footer for provenance. It does not require MCP, CLI, or provider context; local git fallback is
sufficient.
